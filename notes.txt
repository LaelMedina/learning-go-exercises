Float numbers in go:
    - Turns out that using floats in go is a little bit more risking that I thought it would be, specially speaking about
    the currency convertion and the "==" or "!=" comparison operators, since a float is not the exact decimal number 
    but just an aproximation, I have to be careful when using this data type in order to not mess up currency convertions
    or doing comparisons that might not work as I expect, a bug like this in a production codebase could be hard to spot
    and also could be really dangerous.

    - GO is a call-by-value language, which means, the built-in functions take in copies of the values and return said copies.
    Way safer than using pointer which might cause unexpected memory erros.c
    - The slice of a slice shares the same memory with the slice it was sliced from, it means, changes in one slice will
    be reflected in the other, to avoid bugs or unexpected slice overrides, it's better to avoid making slices from slices
    or use a three-part slice expression to prevent "append" from sharing capacity between slices.
